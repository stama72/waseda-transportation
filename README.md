# 4-Week Team Development Workshop: Morning Dashboard

このドキュメントは、「毎朝見たいサイト（Morning Dashboard）」をチームで開発するための**完全ガイド**です。
このREADMEを上から順番に読んで実行すれば、プロジェクトの立ち上げから最終発表まで迷わず進めることができます。

---

## 🚀 Step 1: プロジェクトの準備（リポジトリの作成と共有）

まずは、チーム全員が同じコードを触れるように準備します。

### 代表者の作業（1名のみ）
1. GitHub上でこのリポジトリを **Fork（複製）** するか、**Use this template** から新しい自分のリポジトリを作成します。
2. 作成したリポジトリの `Settings` > `Collaborators` から、**チームメンバー全員を招待**します（権限はWriteでOK）。

### メンバー全員の作業
1. 代表者が作ったリポジトリのURLを開き、ローカル（自分のPC）にダウンロードします。
   ```bash
   git clone <代表者が作ったリポジトリのURL>
   cd team-dev-workshop-morning/starter-kit
   ```
2. パッケージをインストールして、開発サーバーを立ち上げます。
   ```bash
   npm install
   npm run dev
   ```
   ブラウザで `http://localhost:5173` が開き、「Good Morning!」という画面が出れば準備完了です！

---

## 💡 Step 2: 企画を詰める（Day 1）

コードを書く前に、チームで「誰のための、どんな朝を作るか」を話し合います。
以下の項目をチームで決定し、紙のノートやFigma、またはこの `README.md` の下に書き込んでください。

1. **ターゲット（ペルソナ）**: 誰の朝を助けるか？（例: 毎朝ドタバタしている大学生、株価が気になる社会人）
2. **解決する課題**: その人は朝、何に困っているか？（例: 傘を忘れる、ニュースを見る時間がない）
3. **使用するAPI**: 課題を解決するために、何のデータを表示するか？（※ `docs/api-catalog.md` を参照）
4. **ワイヤーフレーム（画面設計）**: どこに何を配置するか、手書きのメモや写真で構わないのでレイアウトの設計図を描きましょう。

---

## 🌿 Step 3: チーム開発の進め方（Git Workflow）

チームで開発を行うための「絶対のルール」です。**絶対に `main` ブランチに直接コードを書いてはいけません。**

### 3-1. 開発用のブランチ（分身）を作る
新しい機能（例: 天気予報）を作る時は、必ず最新のコードから自分専用のブランチを作ります。

```bash
# まず最新の状態にする
git checkout main
git pull origin main

# 新しいブランチを作成して移動する
git checkout -b feature/weather-widget
```
※ ブランチ名は `feature/xxx`（機能追加）や `fix/xxx`（バグ修正）のようにわかりやすくしましょう。

### 3-2. コードを書いて変更を Push する
コードを書き終えて動作確認ができたら、自分のブランチをGitHub（リモート）に保存します。

```bash
# 変更したファイルをすべて追加
git add .

# 変更内容にわかりやすいメッセージをつける
git commit -m "feat: 天気APIからのデータ取得処理を追加"

# GitHubに自分のブランチをPushする
git push origin feature/weather-widget
```

---

## 🤝 Step 4: コードの合流（Pull Request と Merge）

自分が書いたコードを、チームの完成品（`main` ブランチ）に合流させる作業です。

### 4-1. Pull Request（PR / プルリク）を作成する
1. GitHubのリポジトリページを開くと、「Compare & pull request」という緑のボタンが出現しているのでクリックします。
2. **プルリクの書き方**: レビュワー（他のメンバー）が確認しやすいように書きます。
   - **何をしたか**: （例: OpenWeatherMapのAPIを叩いて、東京の気温を表示しました）
   - **どうやって確認するか**: （例: 画面中央のWidgetに「24度」と表示されていればOKです）
   - **スクショ**: 画面のスクリーンショットを貼り付けると満点です！

### 4-2. レビューと Merge（マージ）
1. プルリクを出したら、**必ずチームの「他のメンバー」にコードを見てもらいます**。
2. 他のメンバーはコードと動作を確認し、問題なければコメントで **「LGTM（Looks Good To Me）」** と書き込みます。
3. LGTMをもらったら、緑色の **「Merge pull request」** ボタンを押して合流完了です！

### 4-3. マージ後の後片付け（重要！）
自分のプルリクが `main` にマージされたら、ローカルPCのコードも最新に戻します。

```bash
# mainブランチに戻る
git checkout main

# 最新のコードをGitHubから持ってくる
git pull origin main
```
この状態から、次の開発のために再び Step 3-1（`git checkout -b feature/next-feature`）に戻ります。

---

## 🎉 開発サイクルまとめ

1. 企画とデザインを決める (Step 2)
2. 最新の `main` からブランチを切る (Step 3-1)
3. 開発し、`commit` して `push` する (Step 3-2)
4. GitHubでプルリクを作り、LGTMをもらう (Step 4-1)
5. `main` にマージし、ローカルを `git pull` で最新にする (Step 4-3)
6. **2 に戻る！**

エラーが出たり、コンフリクト（競合）が起きて画面が真っ赤になっても焦らないでください。
**「一緒に画面を見て直す」のが、チーム開発の最大の醍醐味です！**
